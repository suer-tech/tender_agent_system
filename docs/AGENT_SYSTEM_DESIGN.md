# AI Agent System Design Doc

> Версия: 1.0 · Обновлено: 2026-04-22 · Owner: tender-agent team

Документ описывает архитектуру, логику и ограничения ИИ-агентной системы для поиска, оценки и анализа российских тендеров (44-ФЗ / 223-ФЗ + коммерческие площадки).

---

## 1. Why — зачем система нужна

### Проблема

Российский рынок закупок — это **~30–40 трлн ₽ в год** (госзаказ + закупки госкомпаний 223-ФЗ + коммерческие площадки). При этом для поставщика-МСП работа с ним выглядит так:

- **Ручной поиск.** 9 крупных площадок, разные интерфейсы, поиск по ключевым словам даёт сотни нерелевантных результатов в день.
- **Слепая подача.** Поставщик видит НМЦК, но не знает типичной скидки на торгах в этой отрасли и регионе — ставит цену наугад или по наитию.
- **Нет due diligence.** Информация о заказчике (жалобы, одностор. отказы, типичные поставщики) разбросана, в одном месте не собирается.
- **Цикл «проверить ТЗ → оценить шанс → подать» занимает 30–60 мин** на один тендер. Для МСП это = штатный специалист по закупкам = 80–150 тыс. ₽/мес.

Существующие сервисы (Контур.Закупки, СБИС, Тендерплан) решают только первую часть — поиск по ключевым словам. Они **не понимают контекст** поставщика, **не делают ценовых бенчмарков**, **не строят risk-профиль**, и не используют LLM для semantic-match по ТЗ.

### Почему именно сейчас

1. **ДИС Минфина открыт с 01.01.2025** — прямой SOAP-доступ к историческим архивам ЕИС (извещения, протоколы, контракты, жалобы, РНП). До этого данные были только через платные агрегаторы или scraping.
2. **LLM понимают длинные ТЗ.** Gemma 3 27b / Grok 4 Fast читают 50+ страниц документации и извлекают структурированные требования.
3. **Снижается стоимость рассуждения.** Бесплатные/копеечные модели через OpenRouter делают per-tender оценку экономически осмысленной даже для массового потока (100+ тендеров в день).
4. **Агенты могут итеративно уточнять** — не просто «искать по слову», а строить диалог с пользователем: «вы ищете IT-услуги в Москве до 5 млн ₽, правильно? а фокус на разработку или сопровождение?».

### Пользовательская ценность

Для поставщика-МСП (B2B клиент):
- **Время**: 30–60 мин/тендер → 2–3 мин на карточку в чате.
- **Деньги**: ценовой бенчмарк подсказывает реальную скидку → точнее ставка в заявке → больше побед при меньшем демпинге.
- **Риски**: предупреждение о «трудных» заказчиках (высокая доля жалоб, расторжений).

Для аналитика/консультанта:
- Доступ к структурированной витрине (1.8 млн строк за апрель 2026 — полный цикл notice → protocol → contract).
- SQL-запросы для отраслевой аналитики (объём рынка по ОКПД2, концентрация, сезонность).

## 2. What — что делает система

### Функциональный перечень

| Возможность | Статус | Источник данных |
|---|---|---|
| Поиск по zakupki.gov.ru (44-ФЗ) | ✅ MVP | Playwright |
| Поиск по 6 коммерческим площадкам | ✅ MVP | Playwright stealth |
| Фильтрация релевантности (LLM-оценка) | ✅ MVP | LLM (OpenRouter) |
| Извлечение требований из ТЗ | 🟡 планируется | LLM + XML из архивов |
| Ценовой бенчмарк по ОКПД2×регион | 🟡 планируется (данные есть) | `eis_analytics.contracts` |
| Risk-карточка заказчика / поставщика | 🟡 планируется (данные есть) | `eis_analytics.complaints`, `unfair_suppliers`, `unilateral_refusals` |
| Сводная карточка тендера | ✅ MVP (веб) | объединение всех источников |
| Ранжирование выдачи | 🟡 частично | скоринг от LLM |
| Telegram-бот | ✅ MVP | — |
| Веб-чат с агентом | ✅ MVP | FastAPI + WebSocket |

### Входы / выходы

**На входе** — либо текст пользователя в чате («IT-услуги в Москве до 5 млн»), либо структурированный поиск (`/search <фраза>` в боте).

**На выходе** — карточки тендеров, каждая содержит:
- Реестровый номер, заказчик, НМЦК, сроки, ссылку
- LLM-оценку релевантности (0–10, с кратким обоснованием)
- **[планируется]** Ценовой контекст: медианная скидка по ОКПД2 за 12 мес
- **[планируется]** Risk-скор заказчика: сколько жалоб/расторжений за период

### 3.5. Оркестрация

**Real-time поиск (синхронный):**
1. Пользователь пишет запрос в чат.
2. Dialog Agent извлекает параметры (ключевые слова, регион, НМЦК).
3. Orchestrator стартует N параллельных парсеров через `concurrent.futures.ThreadPoolExecutor`.
4. Каждый парсер возвращает список тендеров.
5. Relevance Agent (LLM) оценивает каждый по профилю.
6. Фильтр по порогу + ранжирование по score.
7. Карточки отдаются в чат (WebSocket stream).

**Batch пайплайн ЕИС (асинхронный, запуск по расписанию):**
1. `scripts/load_history.py fetch --preset <period>` — скачивает новые архивы.
2. `scripts/load_history.py parse` — парсит XML в витрину.
3. Готовые SQL-вьюшки обслуживают Risk Agent и Bench Agent.

Пайплайн **идемпотентный**: повторный запуск пропускает уже сделанное (статусы в `eis_history.db`, UPSERT в `eis_analytics.db`).

## 4. Risks and guardrails

### 4.1. Галлюцинации LLM

**Сценарии:**
- Модель придумывает «реестровый номер», которого нет.
- Оценивает как «relevant: true» тендер, не связанный с запросом.
- В summary вставляет несуществующие детали (цены, даты).

**Guardrails:**
- **Фактическая информация идёт мимо LLM.** Реестровый номер, НМЦК, ОКПД2, заказчик, ссылки — парсятся детерминированно и передаются в карточку как есть. LLM видит их только как контекст, не как выходные данные.
- LLM возвращает **строго структурированный JSON** (`{relevant, score, summary, reason}`). При `no_json` — fallback на вторую модель.
- **Threshold гейт**: если `score < 5` — тендер не показывается пользователю. Если модели разошлись в оценках (extreme difference) — помечаем как «спорный».
- **Не объединяем** ответы LLM с фактическими данными в одном поле. В карточке видно, где цифра из ЕИС, а где LLM-оценка.

### 4.2. Ошибки извлечения XML из архивов ЕИС

**Сценарии:**
- Namespace-префиксы меняются между экспортами (`ns2:` vs `ns5:`).
- Новые версии XSD-схемы добавляют поля.
- Пустые или неправильно заполненные теги.

**Guardrails:**
- Парсеры работают **по localname**, не по префиксу (устойчиво к изменению namespace).
- Нет предположения о порядке тегов — каждое поле ищется отдельно.
- **Пустое значение → NULL**, не вылет. Парсер сохраняет запись с частичными данными.
- Логирование через таблицу `parse_runs`: фиксируем archives_total, errors, doc_types — видно, если новый тип вдруг перестал парситься.

### 4.3. Дубли и пропуски

**Сценарии:**
- Один тендер на zakupki.gov.ru и на площадке-оператора (например, sberbank-ast) — два объекта.
- Извещение с версиями v1, v2, v3 — три записи вместо одной.
- Релевантный тендер получил низкий LLM-score и не показан.

**Guardrails:**
- **Dedup по реестровому номеру** как первичный ключ. Площадки дают тот же номер.
- **По версии** — хранится последняя версия (UPSERT по `reg_number`, без `version` в PK).
- **False negatives на LLM**: feedback-кнопка «это релевантно» в боте → пересчёт порогов через неделю. Метрика — recall на обратной связи.
- **Ретро-оценка** — раз в сутки прогон LLM по вчерашним отклонённым с порогом `score >= 3` → проверка, не ошибся ли.

### 4.4. Rate limits и доступность источников

**Сценарии:**
- ЕИС блокирует токен (>90 req/min → `code=13`).
- Площадка банит IP за частый парсинг.
- Anthropic блокирует РФ-IP.

**Guardrails:**
- Token bucket (`core/sources/eis/rate.py`) — hard-limit 90/60 + adaptive backoff на `code=13`.
- Playwright: persistent контекст (кэш + куки), умеренные задержки, stealth-патч.
- LLM через OpenRouter (прокси) — не упираемся в geo-блок Anthropic.
- **Нет параллельных fetch** с одним токеном ЕИС — два скрипта друг другу выжгут лимит.

### 4.5. Compliance и персональные данные

**Сценарии:**
- Тендеры содержат ИНН физлиц (ИП, самозанятые).
- Пользователь в чате пишет чувствительные данные → уходят на OpenRouter.
- Токен ЕИС персональный (привязан к ЭЦП пользователя).

**Guardrails:**
- Все данные **публичны** (ЕИС — открытый источник, 44-ФЗ/223-ФЗ публичны по закону). Но обработка ПДн граждан РФ на иностранных серверах — серая зона.
- **Веб-UI и Telegram-бот на российских серверах** (Reg.ru / Timeweb).
- **Токен ЕИС в `.env`**, `chmod 600`, `.gitignore`. Не логируется в выводах.
- **LLM-запросы**: пользовательский текст уходит на OpenRouter. Для sensitive контекста (например, стратегия цены) рекомендовано будет добавить опцию on-prem LLM (GigaChat / YandexGPT).

### 4.6. Human-in-the-loop

Финальные решения всегда за человеком. Агент **не делает автоматических действий** (не подаёт заявки, не скачивает закрытые документы, не расплачивается). Роли:

- **Агент показывает** — карточки, скоринг, бенчмарки.
- **Агент объясняет** — «почему релевантно» из LLM-reason.
- **Человек решает** — участвовать или нет, по какой цене.

Feedback loop:
- Кнопки «👍 интересно» / «👎 мимо» в боте → корректируют порог и профиль.
- Критические ошибки (бот предложил нерелевантное) → лог для ретро-анализа.

## 5. Evaluation — как измеряем качество

### 5.1. Поиск (Retrieval)

**Метрика — Recall@N:**
Из всех релевантных тендеров за день, сколько мы показали в топ-N.

- **Gold standard**: 30–50 запросов размечены вручную. Для каждого есть список «реально релевантных» реестровых номеров.
- **Прогон**: каждую неделю прогоняем gold-set через систему, считаем Recall@10/@20.
- **Целевой показатель**: Recall@20 ≥ 0.8 для «узких» запросов, ≥ 0.9 для широких.

**Метрика — Source coverage:**
Процент дней, когда все N источников ответили хотя бы раз за час (не лежали).
- **Целевой**: ≥ 95% в месяц на источник.

### 5.2. Отбор (Relevance filtering)

**Метрика — Precision@K:**
Из первых K показанных пользователю — сколько он отметил «👍 интересно».

- Логируется автоматически через feedback-кнопки.
- **Целевой**: Precision@10 ≥ 0.7.

**Метрика — Agreement между LLM-runs:**
Один тендер прогоняем через основную и fallback модель. Разногласия выше threshold → под подозрение.

### 5.3. Извлечение данных из архивов

**Метрика — Field coverage:**
Доля записей, в которых ключевое поле заполнено.
- `notices.max_price` — 100% ✅
- `contracts.price` — 99% ✅
- `contracts.supplier_inn` — 79% (остальные — физлица без ИНН) ✅
- `protocols.final_price` — 86% ✅

Цель — не падать ниже этих значений при добавлении новых периодов (регрессионный тест).

**Метрика — Field accuracy:**
Random sample 100 записей → ручная сверка с `zakupki.gov.ru/card/...`.
- Точность `max_price`, `sign_date`, `supplier_inn` должна быть ≥ 99%.
- Точность `contract_subject` (текст) — human review, subjective.

### 5.4. Полезность аналитики

**Метрика — Bench accuracy:**
По тендерам, где есть полная цепочка `notice → protocol → contract` (сейчас ~7k записей):
- Наша прогнозная скидка = медиана по прошлым контрактам (OKPD2 × регион).
- Реальная скидка = (1 − contract.price / notice.max_price).
- **MAPE** между прогнозом и реальной.
- **Целевой**: MAPE < 20% для «часто встречающихся» комбинаций (OKPD2 × регион с ≥10 контрактами).

**Метрика — NPS по фиче:**
- В боте после показа бенчмарка — «Помогло ли при оценке цены?» (5-балльная).
- Цель — 7+ из 10.

### 5.5. Performance

| Параметр | Целевой | Текущий |
|---|---|---|
| Latency realtime-search (p95) | < 10 сек | не измерено |
| Latency LLM-оценки 1 тендер (p95) | < 3 сек | не измерено |
| Время парсинга месячного периода ЕИС | < 30 мин | 18 мин ✅ |
| Время скачивания месяца ЕИС (10 регионов) | < 6 часов | 4.5 часа ✅ |
| Uptime web | ≥ 99% | не измерено |

| Диск (архивы ЕИС, 1 месяц) | ~15 GB | На VPS с запасом 25 GB |
| Итого | **< $10/мес** | Пока в free-режиме LLM |
